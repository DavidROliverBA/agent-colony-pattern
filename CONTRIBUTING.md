# Contributing to the Agent Colony Pattern

This is v1.0 — a foundation, not a conclusion. It is published to be challenged, refined, and built upon. The worst outcome is not that the argument is wrong. It is that the gaps go unnamed and the same mistakes get made again, one more time, with agents.

## Three Ways to Contribute

### 1. Challenge the Argument

If you think a principle is wrong, a gap is mis-identified, the literature review missed something important, or a claim is unsupported — open an issue. Use the [Challenge template](.github/ISSUE_TEMPLATE/challenge.yml).

When challenging:
- Cite the specific section you disagree with
- State your counter-claim clearly
- Provide evidence at the highest grade you can support (see Evidence Grades below)
- Apply mandatory dissent to your own argument before submitting — what is the strongest case against your challenge?

### 2. Propose Refinements

Corrections, clarifications, extensions, and editorial improvements are welcome as pull requests. Use the [Refinement template](.github/ISSUE_TEMPLATE/refinement.yml) to discuss before submitting if the change is substantial.

PR format:
- **Title** — descriptive, starting with what changed (e.g. "Clarify overlap index thresholds in Section 4")
- **What changed** — the specific text or structure modified
- **Why** — the reasoning, with evidence where applicable
- **What it affects** — other sections, diagrams, or documents that may need updating

### 3. Build a Reference Implementation

The pattern is conceptually grounded but practically untested. If you build a real implementation — even a partial one — link it. It will be added to the roadmap in README.md. What matters is that the work is documented, the limitations are honest, and the findings (positive and negative) are shared.

Contact via GitHub issues or PR with a summary of what you built and where it lives.

## Review Criteria

The paper's own epistemic framework applies to its own development. Contributions are evaluated against the same standards the pattern asks of agent colonies.

### Evidence Grades

| Grade | Meaning | Required for |
|-------|---------|--------------|
| **Empirical** | Directly observed, reproducible | Changing a specification threshold or mechanism |
| **Corroborated** | Observed multiple times across contexts | Adding a new principle or extending the gap analysis |
| **Theoretical** | Logically sound, not yet empirically tested | Extending the argument, proposing new mechanisms |
| **Inherited** | From external sources, not yet validated here | Pointing to relevant prior work or standards |
| **Anecdotal** | Single observation, not yet reproduced | Opening a discussion, flagging a potential issue |

Anecdotal evidence is enough to open an issue. Constitutional changes to the specification require corroborated or higher.

### Mandatory Dissent

For PRs that change a principle, gap, or architectural mechanism: before submitting, articulate the strongest counter-argument against your own change. Include it in the PR description. A contribution that has not been tested against its own best objection is not ready for review.

This is not a hazing ritual. It is how the paper asks agent colonies to reason. The standard applies here too.

### Honest About Limitations

If you do not know something, say so. If your evidence is thin, say so. If your implementation only partially validates the pattern, say so. Overstating certainty is worse than acknowledging uncertainty.

## Code of Conduct

Direct communication is preferred over performative communication. Disagree with arguments, not people. If something is wrong, say what is wrong and why. If you do not know, say you do not know. There is no prestige in being right and no shame in being corrected — both advance the work.

Behaviour that treats contributors as obstacles rather than collaborators will result in removal from the discussion.

---

## What This Repository Is Not

This is not a framework, platform, or product repository. There is no code to contribute. The contributions that matter are intellectual: better arguments, stronger evidence, working implementations, and honest critiques.

The pattern will be wrong in places. The gaps will be mis-scoped in places. The mechanisms will need calibration. These are features of v1.0, not bugs to be embarrassed about. The point is to be wrong in the open, so the corrections can happen in public.
