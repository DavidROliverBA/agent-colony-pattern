# Feedback

Peer review and critique captured as it arrives. One file per review round, dated. Reviewers are attributed by name where permission is given, otherwise anonymised.

## Convention

Each file follows this shape:

```markdown
# [Title] — [Date]

**Reviewer:** [Name or "anonymous peer review"]
**Round:** [number]
**Scope:** [what was reviewed]
**Version reviewed:** [repo version at time of review]

## Summary
[2-3 sentences on what the review said at a high level]

## Key points raised
[bulleted list of specific feedback]

## Verdict
[the reviewer's conclusion]

## What changed in response
[specific repo changes, with commit SHAs or version tags]

## What has not changed and why
[feedback that was heard but not acted on, with the reasoning]
```

The last two sections are the important ones. They make the feedback a closed loop rather than a suggestion box. A reviewer can come back in six months and see exactly what happened to what they said.

## Why negative feedback is kept

The manifesto asks for interrogation, not agreement. A knowledge base that only captures praise is a marketing page with good manners. Every file in this folder includes at least one sharp objection that moved the pattern forward. Where objections were rejected, the reasoning is captured too — because "we heard it and decided not to act" is a different outcome from "we did not hear it at all".
