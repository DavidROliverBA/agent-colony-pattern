# Teaching Colony — External Review, 2026-04-14

> **What this document is.** A substantive critique of the Teaching Colony example after v1.6.1 shipped. The reviewer ran the example, read what the README promised, compared it to what the code actually did, and found several real bugs plus one deep structural issue. The review is preserved here alongside the [design spec](design-spec.md) and [implementation plan](implementation-plan.md) because it is the kind of honest feedback the repository's knowledge-base pattern is built for — and because the v1.7.0 *Honest Teaching Colony* release is a direct response to it.
>
> **Provenance.** Received 2026-04-14 after v1.6.1 was tagged. Reproduced verbatim below.
>
> **Response.** v1.7.0 addresses every finding. See the [v1.7.0 spec and plan](../../../docs/superpowers/specs/) and the CHANGELOG entry for v1.7.0. This document is preserved unchanged as the before-state that v1.7.0 corrects.

---

## The review

I ran the example and dug into the adapters. The critique is substantial — there are several real bugs that a reader running the example will hit immediately. Here it is.

### Headline finding — the demonstration is hollow on the path the README tells you to take

The README's central claim (README lines 79–98) is that Teacher answers a beekeeping question, learns from the corpus, graduates, and then answers an Agent Colony question it "could not have answered at boot time." I ran the example and examined `state/events.jsonl`. Both Teacher answers are identical:

```json
{"topic": "beekeeping",           "answer": {"tokens": 0, "mock": true, "ok": true}}
{"topic": "agent-colony-pattern", "answer": {"tokens": 0, "mock": true, "ok": true}}
```

Neither is a canned answer. Neither is a teaching act. Both are the generic `{"ok": True}` fallback from the end of `_mock_response` (`adapter.py:284`). Why?

Because the Claude Code adapter's mock branches key on `agent_id == "teacher"` and `task == "teach"` (`adapter.py:223`), while `run.py` calls `dispatch_agent("teacher-agent", {"topic": "beekeeping", "question": "..."})` — bare `"teacher"` vs `"teacher-agent"`, and no `task` key at all. The adapter has a lovingly hand-written canned beekeeping answer and a canned Agent Colony answer (`adapter.py:225–243`); both are dead code because the driver never hits them. Same problem for Librarian (`librarian` vs `librarian-agent`, no `task="curate"/compute_coverage"/propose_capability"`), so all Librarian dispatches also fall through to `{"ok": True}`. `concepts_extracted` is always empty. `coverage_score` always lands on the fallback `0.85`.

This is test kabuki: `tests/test_portability.py:191` does use the bare `"librarian"` + `task` shape and therefore fires the canned branches and passes — but `run.py` (the thing the README asks the reader to run) does not. The tests and the demo drift. The v1.6.0 claim "26 passed, 2 skipped, 4 xfailed" (`README:178`) is true and irrelevant, because the tests are not exercising the happy path the README advertises.

This is the single most important fix. Either align `run.py` with the adapter's expected dispatch shape, or align the adapter's expected shape with `run.py`. Until then, the example's headline claim — "Teacher answers a question it could not have answered at boot time" — is false. Teacher answers nothing at all.

### Second finding — the capability never lands in the Mirror

After the run, `colony/mirrors/teacher-agent.yaml` ends up like this:

```yaml
capabilities:
  capabilities:
    - name: teach_beekeeping       # still only this!
      ...
...
capability_add:                    # <-- new top-level key, wrong place
  name: teach_agent_colony_pattern
  description: ...
  maturity: experimental
```

The driver passes `changes={"capability_add": {...}}` to `update_mirror`, and the Claude Code adapter's `_deep_merge` (`adapter.py:320`) treats `capability_add` as a literal new top-level YAML key rather than interpreting it as an instruction to append to `capabilities.capabilities[]`. The pre/post hashes are correct, the evolution_log entry is correct, the `mirror_update` event is emitted. But the new capability is in the wrong place in the YAML and would not be visible to any downstream agent that inspects Teacher's capability list.

A reader who `cat`s the file after running the example, as the README explicitly instructs at line 116, will see a Mirror that says Teacher still only teaches beekeeping — and a mysterious `capability_add:` stanza floating at the bottom. That is a credibility hit on the artefact the README most wants the reader to look at.

This is also why the hollowness of the dispatch path is papered over: even if the dispatch had fired its canned Agent Colony answer, it still wouldn't have been checking whether Teacher actually has the capability, because the mock dispatch doesn't consult the Mirror. But a reader following the README wouldn't know that, because the file on disk visibly does not show Teacher having gained the capability.

### Third finding — the Comprehension Contract is not enforced

`substrate-contract.md` lines 66–81 explicitly states:

> The adapter must:
> 1. Call `classify_action(...)`.
> 2. Reject the call if the change violates the agent's `blast_radius_ceiling` or its `self_evolution_scope.forbidden` list.
> 3. Verify the co-signer's Mirror has a fresh pre-registered policy for the action class, or escalate to the required review regime.
> 4. Compute `pre_state_hash`...

`grep -rn blast_radius_ceiling substrates/` returns zero matches. The Claude Code adapter's `update_mirror` (`adapter.py:304–349`) does none of steps 1, 2, or 3. It deep-merges, hashes, and writes. The Managed Agents adapter in mock mode returns a zeroed-hash stub (`adapter.py:364–373`) and does no enforcement either.

More damning: Teacher's Mirror declares:

```yaml
autonomy.self_evolution_scope.forbidden:
  - "Add new teaching topics autonomously — must go through graduation"
comprehension_contract.blast_radius_ceiling: Inter-agent
```

Meanwhile `mirror_capability_add` has a default blast radius of `Colony-wide` in the classifier (`classifier.py:51`). The capability addition being demonstrated exceeds Teacher's declared blast-radius ceiling, and the adapter cheerfully proceeds. The forbidden entry literally names the thing the example is doing, and nothing catches it.

This is the Comprehension Contract as visual theatre. The three structural invariants the thesis §7 v1.1.1 strengthening rested on — enum, co-sign, audit log — are being co-signed and audit-logged, but the enum/ceiling check is absent. The demonstration was supposed to prove the pattern's governance bites; instead it proves the pattern's governance can be satisfied by rubber-stamps and hash-writing alone. That is the exact failure mode the thesis warned against.

The honest fix is small: `update_mirror` loads the Mirror, reads `self_evolution_scope.forbidden` and `comprehension_contract.blast_radius_ceiling`, calls `classify_action`, and raises `BlastRadiusViolation` or `ForbiddenEvolution` when they fire. Twenty lines. Currently the entire contract-enforcement paragraph of `substrate-contract.md` is fiction.

### Fourth finding — the event count is duplication masquerading as observability

`README` line 159 explains the 42-vs-22 event gap between Claude Code and Managed Agents as "adapter-internal events on top of the driver events… observability richness, not the graduation proof." The framing is generous. Looking at the events by type:

| Driver event | Adapter-internal equivalent |
|---|---|
| `kb_write` ×7 | `kb.written` ×7 |
| `mirror_update` ×1 | `mirror.updated` ×1 |
| `cosign` ×1 | `cosign.granted` ×1 |

Plus 11 `dispatch.complete` events that have no driver equivalent. These are not additional observability. They are the same logical actions being recorded twice, once by `run.py` via `emit()` and once by the adapter's internal `record_event` calls. Any consumer reading the event log would see double-counted KB writes and have to deduplicate by topic + provenance + timestamp. The underscore-vs-dot naming is the only way to tell them apart.

Either the driver emits and the adapter stays silent, or the adapter emits and the driver stays silent — but currently both do, with slightly different naming conventions. That's not a principled design; it's two layers that weren't coordinated. Since the `managed_agents` adapter doesn't emit the internal events, the parity claim breaks not because of a v1.7 gap but because of an inconsistent event ownership model that no document in the repo acknowledges.

### Fifth finding — the portability claim, honestly re-stated

Given the four findings above, what does the example actually demonstrate?

- **Principle 2 (Identity over implementation):** The classifier is byte-identical across substrates (`test_portability.py` asserts this). The Mirror YAML files are byte-identical. These are true and worth keeping. But the claim the classifier is load-bearing assumes the adapters consult it on writes, and they don't.
- **The eight-operation substrate contract:** Mechanically satisfied — both adapters have matching method signatures. But the contract's stated enforcement clauses (`substrate-contract.md` lines 66–81) are not honoured by either adapter, so "the substrate owns Mirror storage" is lip service.
- **Graduation through Comprehension Contract:** Ceremonial. No check would have failed regardless of input.
- **Learning from the corpus:** Does not happen. All Librarian calls return `{"ok": True}`. The corpus files are read to compose provenance strings and are otherwise ignored by the mock adapter.

The honest version of the v1.6.0 portability claim is therefore: "Two adapters with matching method signatures can run a hard-coded nine-step lifecycle driver to completion in mock mode, emitting a stable set of event types and writing a predictable evolution log entry to a YAML file, without any actual governance enforcement and without the agents doing anything the Mirror state would influence." That's still a non-trivial scaffolding achievement, but it is roughly two rungs below what the README claims.

### Smaller issues worth noting

- `README.md:52–53` says "it takes three progressive runs to see everything". Run 2 is advertised as a determinism check and is a good idea, but given the current mock dispatch is a generic passthrough, determinism is trivially true for reasons that have nothing to do with the pattern's properties.
- `README.md:126` claims "~42 events covering 14 unique event types". I counted 14 types and 42 events, so this is accurate — but the number being accurate is the result of the duplication described above, not of the pattern working.
- `README.md:128` admits the adapter writes back into `colony/mirrors/` (git-tracked) and tells readers to `git restore` after running. That's a design smell the README partly owns, but the fix ("copy-on-write or write to `state/`") should really block v1.6 shipping — an example that leaves git-dirty files in the reader's working copy after a no-argument run is a bad onboarding experience. The `--reset` flag wipes `state/` but does not revert the Mirror, so a reader who runs Run 1, then Run 3, then Run 1 again will have their Teacher Mirror monotonically growing with evolution_log entries across runs.
- `colony/corpus/pattern/` contains `manifesto.md`, `specification.md`, `thesis.md` as copies of the repo's top-level files. There's no sync check or comment saying "these are snapshots, re-sync before editing". A reader who edits the top-level files will find the example still citing the old versions silently. A README in `corpus/` saying "snapshot of repo root at v1.6.0; re-sync with `scripts/sync-corpus.py`" would cost 10 minutes.
- `substrate-contract.md:42–45` justifies removing `read_event` because "Event Memory queries are a specialised form of `read_kb` in every substrate we tried." Neither adapter actually implements this unification — `read_kb` in both adapters queries the KB directory, nothing queries the event log. The justification is aspirational, and the "we tried" is load-bearing rhetoric that can't currently be verified.
- `classifier.py:115` `blast_radius = action.get('blast_radius') or ACTION_BLAST_RADIUS.get(...)` — the caller could override the blast radius by passing it in the action dict. Combined with the adapter not validating the caller's classification, a malicious or confused caller could submit `{"class": "mirror_capability_add", "blast_radius": "Local"}` and the classifier would trust it. The Comprehension Contract's "callers cannot classify their own actions" (`substrate-contract.md:59`) is not in fact enforced by the classifier itself — it's only enforced by convention, and the adapter never reviews the classification it gets back. A minimum fix: the classifier should ignore caller-supplied `blast_radius` for action classes that appear in `ACTION_BLAST_RADIUS`.

### What works and is worth keeping

- **The eight-operation contract as a shape.** It's the right abstraction level. Bugs are in the implementations, not the interface.
- **The structural classifier (`colony/logic/classifier.py`)** is pure, table-driven, and ports cleanly. The short-circuit order (Constitutional → Novel → Pre-registered → default) is the right design. Given the caller-supplied `blast_radius` loophole above it needs tightening, but conceptually this is the strongest 150 lines in the example.
- **Two substrates at all.** Even with the Managed Agents adapter mostly mocked/scaffolded, having a second adapter file present forces the author to confront concerns that would otherwise be invisible. The `gaps.md` document is unusually honest about what's real and what's scaffold. That style should be the template for every non-trivial artefact in the pattern.
- **The design-spec + implementation-plan pair in `docs/`.** Keeping the scoping and the five-sub-agent execution plan in-repo is unusual and correct — it turns the example from "here's what was built" into "here's how it was built, and what was traded off". Reviewers can argue with the plan, not just the output.
- **Reset semantics and mock mode.** `--reset` + `--mock` is the right UX for a first-time reader. Offline, no API key, deterministic — all good.

### Priority fix list (in order)

1. **Make the mock dispatch path actually fire.** Either pass `task=` and bare agent ids from `run.py`, or rewrite `_mock_response` to key on suffixed agent ids and driver input shape. Until this is done the example is demonstrably false about its central claim. Cost: 20 minutes. Test: assert that `state/events.jsonl` contains an `agent_serve` event whose `answer.answer` string contains the word "worker" for the beekeeping turn and the word "Mirror" for the Agent Colony turn.
2. **Fix `update_mirror` deep-merge for `capability_add`.** Interpret the change dict as a semantic instruction, not a literal YAML merge. Either introduce a small change-DSL (`{"add_capability": {...}}`, `{"remove_capability": "name"}`) or parse `capability_add` and append to `capabilities.capabilities[]`. Test: after Run 1, `capabilities.capabilities` contains exactly `[teach_beekeeping, teach_agent_colony_pattern]` and there is no top-level `capability_add` key. Cost: 30 minutes.
3. **Enforce the Comprehension Contract clauses the `substrate-contract.md` says are enforced.** `update_mirror` must check `self_evolution_scope.forbidden` and `blast_radius_ceiling`, call `classify_action`, and raise on violation. The current `forbidden` entry in Teacher's Mirror ("Add new teaching topics autonomously") should either be loosened (because graduation is not autonomous) or tightened and the graduation path distinguished from the forbidden path. Either way, the text and the behaviour must agree. Cost: 60 minutes including a new test asserting that an unco-signed capability add raises.
4. **Resolve the dual-event-stream inconsistency.** Pick one layer as the event owner. Recommendation: the adapter owns adapter-internal events (`kb.written`, `mirror.updated`, `cosign.granted`, `dispatch.complete`) and the driver stops emitting `kb_write`, `mirror_update`, `cosign`. That gives Managed Agents a clear gap to close (emit these events too) rather than the current "doesn't bother" state, and it aligns with the `substrate-contract.md` design that puts state ownership in L4. Cost: 45 minutes. Side benefit: the 42-vs-22 gap becomes a real, testable v1.7 parity target rather than an accounting note.
5. **Move Mirror writes out of `colony/mirrors/`** and into `state/mirrors/` (or make them copy-on-write against a `state/mirrors/` overlay). Currently running the example dirties a git-tracked file. This is in the README's "Known follow-up (v1.7+)" but it's a reasonable v1.6.x hotfix — a first-time reader running the example should not have to `git restore` anything. Cost: 45 minutes.
6. **Gate caller-supplied `blast_radius` in the classifier.** For any action class listed in `ACTION_BLAST_RADIUS`, the table value wins; caller override only applies to action classes the table doesn't know about. Cost: 5 minutes. Test: passing `{"class": "mirror_capability_add", "blast_radius": "Local"}` still returns `Colony-wide`.

### Overall

This is the most ambitious artefact in the whole `agent-colony-pattern` repo, and the structure of the artefact is right: two substrates, a driver that doesn't know which one it's using, a contract abstract class, a classifier that's a pure function, a design spec in the repo next to the code, and a gaps report that owns what's missing. That's the correct shape for a portability proof. What's wrong is the wiring. The driver, the adapter mock, and the Mirror merge logic were written by people (or sub-agents) who didn't run the end-to-end path together, and the test suite passes because the tests exercise the adapter's mock branches directly rather than running the full lifecycle the README promises.

The good news is that every finding above is cheap to fix and none of them require a redesign. Three to four hours of focused work closes findings 1, 2, and 3, after which the example actually demonstrates what it claims — and the thesis has its first concrete artefact that a sceptical reader can't dismiss as prose. Leave the findings unfixed and the example becomes the highest-risk item in the repo: a 4,300-line artefact whose headline demonstration doesn't hold up under a single run-and-inspect, on a project whose credibility rests on the author's v1.1.1-era willingness to own what's broken. The cure is to apply the same honesty to the example that the thesis now applies to itself.
